# 🏗 Smartoo Workspace Infrastructure

Infrastructure as Code (IaC) для развертывания и управления изолированными контейнеризованными средами разработки (`smartoo-workspace-dev`), а также конфигурацией защищенных SSH-туннелей через Cloudflare Zero Trust.

## 🗄 Структура репозитория
* `docker/` — Конфигурации для развертывания инфраструктурных сервисов.
* `scripts/` — Скрипты инициализации
* `templates/` — Эталонные конфигурации для клиентских машин.

## 🌐 Инфраструктура и Экосистема Проектов

Текущая инфраструктура распределена по следующим ключевым направлениям и доменным зонам:

### 🛠 Базовая экосистема (Smartoo)
* **`code.smartoo.dev`** — Облачная среда разработки (Workspace IDE). Полностью изолированный Docker-контейнер с доступом через защищенный туннель Cloudflare.
* **`git.smartoo.dev`** — Локальный сервер контроля версий (Gitea). Выступает единым источником истины для конфигураций инфраструктуры с автоматическим зеркалированием (Push Mirror).
* **`team.smartoo.dev`** — Пространство для командного взаимодействия и координации проектов.

### 🌍 Корневые узлы (Core Nodes)
* Базовые VPS-узлы, обеспечивающие фундамент для изолированных виртуальных машин, транзитных туннелей и сервисов маршрутизаци.

## 🚀 Развертывание локального Git-сервера
В качестве основного источника истины используется Gitea. Сервер развернут с использованием Dokploy и доступен по адресу: **[https://git.smartoo.dev](https://git.smartoo.dev)**.

Репозиторий настроен на автоматическое зеркалирование (Push Mirror) в облако (GitHub/GitLab) для обеспечения резервного копирования при сохранении полного суверенитета инфраструктуры.

## 🛠 Решение известных проблем (Troubleshooting)

### 1. Блокировка WebSocket со стороны систем DPI (ТСПУ)
**Симптом:** Соединение моментально сбрасывается с ошибкой `read: connection reset by peer` или `Connection closed by UNKNOWN port 65535` при успешной HTTP-аутентификации.
**Причина:** Системы глубокого анализа трафика разрывают долгоживущие WebSocket-соединения утилиты `cloudflared`.
**Решение:** Заворачивание трафика Cloudflare внутрь зашифрованного прокси-соединения (Xray/sing-box) с помощью системной переменной `ALL_PROXY`. Указывается локальный порт (например, `10808`). 
*См. шаблон `templates/ssh_config_example`.*

### 2. Ошибка локального агента ключей (GNOME Keyring / Pop!_OS)
**Симптом:** Ошибка `sign_and_send_pubkey: signing failed... agent refused operation`.
**Причина:** Системный `ssh-agent` блокирует использование ключа или пытается отправить неверные идентификаторы.
**Решение:** Жесткое игнорирование фонового агента путем добавления параметра `IdentitiesOnly yes` в конфигурацию хоста.

### 3. Ошибка Permission denied внутри контейнера
**Симптом:** Сервер отклоняет публичный ключ, несмотря на успешную работу туннеля.
**Причина:** Некорректные права доступа к директории `.ssh` при монтировании Docker-томов.
**Решение:** Владельцем файлов должен быть целевой пользователь (не `root`), а права должны быть строго ограничены (700 на папку, 600 на ключ).
*См. скрипт `scripts/fix_container_keys.sh`.*
